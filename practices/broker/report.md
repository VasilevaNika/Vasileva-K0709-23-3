# Отчет: сравнение RabbitMQ и Redis как брокеров сообщений

##Конфигурация стенда
- Producer: `producer.py`
- Consumer: `consumer.py`
- Оркестратор тестов: `run_benchmarks.py`
- Ограничения ресурсов для обоих: CPU `2.0`, RAM `2GB`

## Параметры экспериментов
- Размер сообщения: `128 B`, `1 KB`, `10 KB`, `100 KB`
- Rate: `1000`, `5000`, `10000 msg/sec`
- Duration (sec): `20 sec`

## Выводы

1. Redis показал больше пропускную способность на всех тестах при равных size и duration. Также он был ближе к целевому rate, а RabbitMQ зачастую сильно отставал от целевой пропускной способности.

2. Оба брокера сохранили нулевые потери и ошибки при увеличении размера сообщений. При этом у RabbitMQ p95_latency росла заметнее, соответственно Redis лучше переносит увеличение размера сообщения.

3. Будем считать, что деградация, это ситуации, когда потери или ошибки больше нуля или реальная пропускная способность стала менее 90% от целевой.
RabbitMQ начал деградировать при нагрузке 1000 msg/sec и размере сообщения 128 байт. 
Redis начал деградировать при нагрузке 5000 msg/sec и размере сообщения 100 Кб.

4. Если нужен стабильный брокер с предсказуемой доставкой, то лучше использовать RabbiMQ. Если важнее минимальные задержки, то лучше использовать Redis.
